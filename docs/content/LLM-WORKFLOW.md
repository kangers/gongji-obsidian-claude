# LLM Workflow — 公基知识库维护指南

> 本文件是 LLM Wiki 模式的 **Schema 层**。它告诉 AI 助手如何维护 `wiki/` 目录下的知识库。
> 参考: [LLM Wiki](LLM Wiki)

---

## 一、知识库结构

```
📁 公共基础知识库/
├── 📁 raw/                      ← 第1层：原始来源索引
│   └── index.md                 ← 原始文件清单（PDF 实际在各目录中）
│
├── 📁 wiki/                     ← 第2层：AI 维护的知识库
│   ├── index.md                 ← 内容目录（AI 自动更新）
│   ├── log.md                   ← 操作日志（AI 自动追加）
│   ├── 总览.md                  ← 知识库总览
│   ├── 📁 概念/                 ← 抽象知识体系（政治、法律、管理、经济、公文等）
│   ├── 📁 实体/                 ← 具体对象（宪法、民法典、公务员法等）
│   ├── 📁 来源/                 ← 每个来源文件的摘要与提取状态
│   └── 📁 策略/                 ← 备考策略、高频考点、易错点
│
├── 📁 政策解读/                 ← 政策原文（Markdown）
├── 📁 辅导班题目/               ← 模拟试卷（Markdown）
├── 📁 练习题+解析/              ← 按科目分类的练习题 PDF
├── 📁 公基大默写/               ← 默写练习 PDF
├── 📁 公基必刷1600/             ← 必刷题 PDF
│
├── 📄 *.pdf                     ← 核心教材PDF（不可变）
├── 📄 LLM-WORKFLOW.md           ← ⬅️ 本文件（Schema）
└── 📄 LLM Wiki.md               ← 模式说明文件
```

## 二、页面格式规范

### 2.1 Frontmatter（YAML）

每个页面必须以 YAML frontmatter 开头：

```yaml
---
创建时间: 2026-06-08        # 首次创建日期
更新时间: 2026-06-08        # 最后修改日期
分类: 概念/实体/来源/策略/索引/日志  # 页面类型
标签: ["标签1", "标签2"]     # 相关标签
状态: 待完善/活跃/已归档     # 当前状态
来源: ["来源1", "来源2"]     # （可选）信息来源
---
```

### 2.2 页面命名规范

- 中文命名，简洁明了
- Wiki-link 用 `[显示名](wiki/分类/页面名)` 格式
- 概念页面使用通用名词（政治、法律、管理）
- 实体页面使用具体名称（宪法、民法典、公务员法）

### 2.3 概念页面结构

概念页面用于抽象知识体系，结构如下：

```
# 概念名

> 一句话概述

## 子领域
- 子领域1 — 说明
- 子领域2 — 说明

## 核心知识点
（知识点列表）

## 相关来源
- [xxx](wiki/来源/xxx)

## 相关链接
- [xxx](wiki/xxx)
```

### 2.4 实体页面结构

实体页面用于具体对象（法律、人物、机构等）：

```
# 实体名

> 一句话概述

## 核心要点
（关键内容）

## 重要细节
（详细信息）

## 相关概念
- [xxx](wiki/概念/xxx)

## 相关来源
- [xxx](wiki/来源/xxx)
```

### 2.5 来源页面结构

来源页面用于追踪原始资料的处理状态：

```
# 来源名称

> 一句话说明

## 文件信息
- **原始路径**: xxx
- **类型**: PDF/Markdown/其他

## 提取状态
- [ ] OCR 提取
- [ ] 摘要完成
- [ ] 已整合概念页面
- [ ] 已整合实体页面
```

## 三、工作流

### 3.1 Ingest（摄入新来源）

当用户说"摄入 [文件]"或"处理 [来源]"时：

1. **读取来源** — 读取文件内容（PDF 使用 Read 工具）
2. **讨论要点** — 与用户讨论关键发现
3. **写摘要页** — 在 `wiki/来源/` 创建或更新来源摘要页
4. **更新目录** — 更新 `wiki/index.md` 中的来源条目
5. **更新概念页** — 更新相关概念页面的知识点
6. **更新实体页** — 更新相关实体页面的内容
7. **更新策略页** — 如果有新的高频考点/易错点，更新策略页面
8. **记录日志** — 在 `wiki/log.md` 追加新条目（格式见下方）
9. **更新来源状态** — 将来源页面的提取状态标记为完成

**重要原则**：一次处理的来源可能影响 5-15 个 wiki 页面，要逐一检查和更新。

### 3.2 Query（查询问题）

当用户问关于公基考试的问题时：

1. **读目录** — 先读取 `wiki/index.md` 定位相关页面
2. **读相关页** — 读取匹配的 wiki 页面
3. **综合回答** — 基于 wiki 内容综合回答，引用来源
4. **可选：存档答案** — 如果需要，将好的问答转化为新的 wiki 页面

**重要**：优先使用 wiki 回答问题，而不是直接去读原始 PDF。如果 wiki 内容不足以回答，再去读原始来源。

### 3.3 生成真题

当用户要求生成模拟试卷时：

1. 参考 `wiki/总览.md` 中的考试结构（100单选+20多选+20判断）
2. 参考 `wiki/策略/高频考点.md` 确定考点分布
3. 参考现有模拟卷格式（`辅导班题目/模拟1.md`）
4. 生成试题并保存到 `辅导班题目/`
5. 记录到 `wiki/log.md`

### 3.4 Lint（健康检查）

当用户说"检查 wiki"或"lint"时：

检查以下内容：
- **矛盾** — 不同页面间是否有不一致的陈述
- **过时** — 是否有被新来源取代的旧信息
- **孤页** — 没有入链的 orphan 页面
- **缺失** — 来源中已提取但尚未创建对应 wiki 页面的概念
- **缺失链接** — 明显相关的概念/实体之间缺少交叉引用
- **数据缺口** — 可以通过搜索补充的内容

### 3.5 日志格式

`wiki/log.md` 的每个条目使用以下格式：

```
## [YYYY-MM-DD] 操作类型 | 简短标题

- 操作1：做了什么
- 操作2：更新了哪些页面
```

操作类型：`ingest`（摄入）、`query`（查询）、`lint`（检查）、`init`（初始化）、`generate`（生成）

## 四、跨页面一致性规则

1. **同步更新**：当更新一个概念页面时，检查所有引用它的页面是否需要同步更新
2. **不得矛盾**：如果两个页面对同一事实的描述不一致，两者都必须修正，并在 log 中注明
3. **交叉引用**：概念和实体页面之间应建立双向 Wiki-link
4. **状态标记**：不完整的页面保持 `状态: 待完善`，完成后改为 `状态: 活跃`
5. **来源追踪**：每条重要知识应标记来源（通过"相关来源"章节）
